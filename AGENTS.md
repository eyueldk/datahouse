## Learned User Preferences

- Use `bun add` and `bun remove` for dependency changes instead of hand-editing dependency blocks.
- Prefer object-style function parameters named `params` (for clarity and future extensibility).
- Always use spinal-case for filenames and mapping keys. In `@datahousejs/server` Drizzle schemas, use `uuid().defaultRandom()` for primary keys and foreign keys (not prefixed `nanoid` varchar IDs); use `nanoid` only for non-row identifiers when needed (e.g. storage object keys).
- Keep modules function-oriented; export standalone functions instead of service objects with methods. In `@datahousejs/core`, **`createExtractor` requires a `config` block** (schema + `create`); sources are created with **required** `config` in the APIâ€”do not add a â€śmanual key onlyâ€ť path. Default extractor config/cursor generics to `{}` rather than `unknown`.
- Validate environment variables at module load time so failures happen fast.
- When the user provides a plan with existing todos, implement it directly, do not edit the plan file, and update the existing todo statuses in order.
- Use `uv` for Python package management and command execution (`uv add`, `uv run`); do not generate application migrations unless explicitly requested. **Never** hand-write or edit Drizzle migration SQL, `packages/server/migrations/meta/_journal.json`, or migration snapshots, or run `drizzle-kit` unless the user explicitly asks for that step in the task.
- When improving declaration emit or â€śisolated declarationsâ€ť behavior, distinguish **tsdown DTS settings** (e.g. `dts.oxc` in `packages/datahouse`) from **`compilerOptions.isolatedDeclarations`** in a package **`tsconfig.json`** â€” the user may want one without the other. Prefer adjusting **`packages/datahouse/tsdown.config.ts`** (and related build entrypoints) over editing package **`tsconfig.json`** when the goal is a passing **`bun run build`** without broad tsconfig churn, unless the user explicitly asks for tsconfig **`isolatedDeclarations`**.
- Prefer public or keyless HTTP APIs for examples and local demos when the user wants zero API keys or minimal credential setup; do not catch and rethrow errors unless adding handling/processing, otherwise let errors propagate naturally. Avoid wrapping logic in `try/catch` by defaultâ€”use it when something must run on failure (e.g. resource cleanup), when recovering or translating errors at a deliberate boundary, or when the user explicitly asks for handling; otherwise let failures propagate. In `@datahousejs/server` Elysia routes, skip redundant route-level `try/catch` on simple delegates; for deletes prefer **idempotent** services, respond with **200** and plain **`{ success: true }`** (no `as const`), and do not map â€śalready goneâ€ť to **404**. Prefer concise READMEs (scope, quickstart, links) over long narrative when editing READMEs in this repo.
- For Docker or compose workflows, do not run `docker build` / `docker compose up --build` unless the user asks; they often build images locally themselves. Log clear step-by-step progress during long-running container or image operations unless the user opts out.
- In `@datahousejs/client`, prefer inferred return types instead of duplicating response types; do not clamp or default pagination params such as **`limit`**/**`offset`**â€”pass values through and let the server enforce limits/defaults. Keep the published package surface minimalâ€”only **`createClient`** and type **`DatahouseClient`** on **`exports["."]`**; avoid extra type barrels in **`package.json`**. Use **`Record`** (not **`Row`**) in client-facing names for datalake/datawarehouse items (**`DatalakeRecord`**, **`DatawarehouseRecord`**). Put internal helpers under **`utils/`** with **`index.ts`** (barrel); import from **`utils`**, not a single **`util.ts`**. Prefer **interface + factory** for the client: the factory annotates the return interface; keep the implementation object unannotated so types infer from the factory and **Elysia `Server` / Eden treaty** stay out of the public **`.d.ts`** shape. In `@datahousejs/server` Elysia routes, name reusable schema symbols with PascalCase and a `Request` or `Response` suffix; declare `response` schemas and return with `status(code, payload)` when wiring endpoints.
- Do not maintain hand-written `.d.ts` files alongside `.ts` sources; use `.ts` only unless the build emits declarations (e.g. published `dist/*.d.mts`). For third-party libraries, prefer exported types from official docs or Context7 instead of broad `as` casts.

## Learned Workspace Facts

- This repository is a Bun monorepo using Prettier conventions (double quotes and semicolons).
- server package name is `@datahousejs/server`.
- The `datahouse` CLI lives in `packages/datahouse` (`bin`: `dist/cli.mjs`); commands are split into `src/commands/{command}.ts`; default config discovery checks `index.js`, `index.ts`, `src/index.js`, then `src/index.ts`.
- `packages/example` is the reference book integration (Open Library and Wikidata over HTTP, no API keys; Wikidata uses `wikibase-sdk` for SPARQL URLs). Code lives under `packages/example/src/extractors/` and `packages/example/src/transformers/`. Scripts `dev` and `studio` `cd` to the monorepo root and run `bun --watch ./packages/cli/src/index.ts` (serve vs studio) so the CLI hot-reloads; Bun only watches files under the process cwd, so running `bun --watch` solely from `packages/example` does not reliably watch sibling packages. `start` invokes the packaged `datahouse` CLI for non-watch runs.
- Only the npm package `datahouse` is published (`@datahousejs/*` are internal); it builds with `tsdown` (`packages/datahouse/tsdown.config.ts`), emits library entries and `dist/cli.mjs`, copies `@datahousejs/server/migrations` into `dist/migrations`, copies `@electric-sql/pglite/dist/pglite.data` and `pglite.wasm` next to `cli.mjs` (PGlite loads them via `import.meta.url`), and copies the built studio app from `../studio/dist` into `dist/studio`. Do **not** list `@datahousejs/studio` as a `workspace:*` devDependencyâ€”`bun publish` cannot resolve workspace versions for private packages; instead map `@datahousejs/studio/server` via `tsconfig` paths to `../studio/src` (build studio before `datahouse` when packaging). Container `Dockerfile`(s) set `NODE_ENV=production` in image stages.
- server server is Elysia-based: loads config and runs DB migrations on startup; background work uses bunqueue queues (embedded or TCP to a bunqueue server); default HTTP port is `2510` (`datahouse serve` / `startServer`). Studio UI is bundled under `dist/studio`; `datahouse studio` defaults to port `2511`.
- **Studio and `@datahousejs/client`:** the Studio **browser** bundle must not instantiate or call `@datahousejs/client` to reach the Datahouse API (no shared `VITE_*` client module for API calls). TanStack Start **`createServerFn`** handlers in **`packages/studio/src/lib/server-functions.ts`** own the Datahouse client (`DATAHOUSE_URL` / `process.env.DATAHOUSE_URL`, default `http://localhost:2510`); use **GET** server functions for queries and **POST** for mutations. Studio API reads should use React Query domain hooks under **`packages/studio/src/hooks/`** rather than route loaders; configure React Query with TanStack Start's router-context **`QueryClient`** plus root **`QueryClientProvider`** pattern. Generated server-function hook factories should call **`useServerFn`** and accept hook-style **`useOptions`** callbacks (callbacks may call hooks such as **`useQueryClient`**). When defining new server functions that don't need input validation, use the `asServerFn({ method?, fn })` utility from `packages/studio/src/utils/as-server-fn.ts` instead of raw `createServerFn`.
- The shared Studio **`DataTable`** (`packages/studio/src/components/ui/data-table.tsx`) enables **client-side pagination** by default (TanStack Table pagination model, default page size **25**); server/API-backed routes should prefer server pagination and pass **`clientPagination={false}`** so pagination is not applied twice. It uses truncated, horizontally scrollable, resizable columns by default, with `meta.noTruncate` for explicit opt-outs.
- server route plugins live under `packages/server/src/routes/`; HTTP API is mounted under `/api` (extractors, sources, runs, datalake, datawarehouse, transformers, etc.). **`DELETE`** handlers for resource rows are expected to be **idempotent** at the service layer (missing row is a no-op) so routes can return **200** with **`{ success: true }`** without treating â€śnot foundâ€ť as a separate client error for delete.
- Server database: Drizzle with PostgreSQL when `DATABASE_URL` is set; otherwise embedded PGlite under `DATA_DIR/pglite`; migrations live in `packages/server/migrations/` (see `packages/server/drizzle.config.ts`). Table definitions are per-file under `packages/server/src/schemas/`; relations stay in `schemas/index.ts`; services import the specific table modules they need rather than `import *` from the schema barrel. Row primary keys and FKs are `uuid` with `defaultRandom()` (`gen_random_uuid()`), not prefixed `nanoid` strings. For `superjsonb` columns that need narrower TypeScript than `unknown`, prefer `.$type<...>()` on the column over introducing a second custom column type.
- Data pipeline model is `sources â†’ runs â†’ datalake â†’ datawarehouse`: **`datalake`** rows hold extracted payload per `run_id`, `source_id`, and `extractor_id`; **`datawarehouse`** rows hold transformed output per `run_id`, `datalake_id`, `transformer_id`, and `collection`. Both tables include a **`metadata`** jsonb column (SuperJSON via `superjsonb`) for arbitrary string-keyed values from extractors and transformers. The **`files`** table stores blob metadata only; **`file_references`** links a file to exactly one datalake or datawarehouse row per reference, allowing one file to be referenced by multiple records. `syncFileLinks` inserts/deletes reference rows, the files API returns `references`, and rows with no references are trash/orphans cleaned by **`orphan-files-cleanup.queue`** / **`deleteFile`**.
- `@datahousejs/client` **`package.json` `exports`** only **`"."`**. The entry exports **`createClient`** and **`DatahouseClient`** (type); **`createDatahouseClient`** and **`create*Client`** factories under **`src/apis/`** compose **`datalake`**, **`extractors`**, **`runs`**, **`sources`**, **`transformers`**, and **`datawarehouse`**. Eden/treaty helpers live under **`src/utils/`**. Client-facing tables use **`DatalakeRecord`** / **`DatawarehouseRecord`**. Monorepo apps import additional types from **`packages/client/src/...`** via **`compilerOptions.paths`** when needed (not via extra publish entries).
